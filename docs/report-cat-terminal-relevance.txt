1) OBIETTIVO
Analizzare il terminalino React "catTerminalInput (cat@facts)" e migliorare la logica API per restituire frasi realmente connesse all'input utente, con routing lingua IT/EN, robustezza edge-case e test.

2) ANALISI INIZIALE DEL CODEBASE
2.1 Componente terminalino
- File: src/components/Motivate.jsx
- Stato principale:
  - frase: stringa mostrata nella modale
  - keyWord: testo input utente
  - fetching: stato loading spinner
  - modalRest: persistenza modale al passaggio mouse
- Handler principali:

2.2 Endpoint e algoritmo originale
- Endpoint: chiamata diretta client-side a https://meowfacts.herokuapp.com/?count=100
- Algoritmo:
  - split input in parole
  - filtro con includes sulle frasi remoti
  - selezione random della frase filtrata
- Limiti:
  - nessun ranking semantico
  - nessuna API interna controllata
  - nessun language routing robusto
  - scarsa resilienza a typo/plurali/punteggiatura

3.2 Data source

3.3 Caching e costi/latency
- Cache facts remoti in-memory (TTL 10 minuti)

- Beneficio: meno roundtrip rete, risposta più stabile e veloce

3.4 Rate-limit e logging minimo
- Rate-limit in-memory: 30 richieste/minuto per IP
- Logging minimale:
  - query
  - lingua risposta
  - reason del ranking/fallback
  - numero totale facts

4) RELEVANCE ENGINE (2 FASI)
File: src/utils/catFactsEngine.js

4.1 Fase A - Candidate Retrieval
Implementato un retrieval lessicale robusto prima del ranking:
- normalizeText:
  - lowercase
  - rimozione accenti combinati
  - rimozione punteggiatura
  - normalizzazione spazi
- tokenize:
  - stopwords IT/EN
  - stemming leggero (suffix trimming)
- typoSoftMatchScore:
  - Levenshtein limitato (max 1 errore) per typo leggeri
- retrieveCandidates:
  - priorità pool lingua (IT per query IT, EN per query non IT)
  - se pochi risultati, fallback pool misto
  - top-N candidati (MAX_CANDIDATES = 25)

Commento codice inserito:
- "Weighted final score keeps lexical precision while adding semantic signal."
- Questo rende esplicita la scelta progettuale: precisione lessicale + segnale semantico.

4.2 Fase B - Ranking
Ranking multi-segnale su candidati:
- BM25 lightweight (term relevance classica IR)
- cosine similarity su trigrammi di caratteri (proxy semantico leggero)
- jaccard similarity tra set token query/documento
- score finale pesato:
  finalScore = bm25*0.6 + semantic*0.3 + jaccard*0.2 + lexical*0.1 + typo*0.1
- ordinamento discendente e scelta top-1

5) LANGUAGE ROUTING
5.1 Language detection
- Funzione: detectLanguage(input)
- Heuristics:
  - stopwords IT/EN
  - suffix/hints italiani (es. -zione, -mente, gatti, fusa)
  - caratteri accentati
- tie-break: default EN per preservare comportamento storico

5.2 Politica risposta lingua
- Se query italiana:
  - priorità facts italiani
  - se top match EN -> fallback traduzione lightweight IT
- Se query non italiana:
  - risposta inglese

5.3 Fallback translation
- Funzione: lightweightTranslateEnToIt
- Dizionario minimo e prefisso "Fatto sui gatti:"
- Scelta voluta:
  - nessuna dipendenza esterna
  - latenza/costi bassi

Commento codice inserito:
- "Fallback translation is intentionally lightweight to avoid external API latency/cost."
- "For Italian input, keep language guarantee even when only EN data matches."
